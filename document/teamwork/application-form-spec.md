## 1. 개요

F-Lab의 지원서 폼을 참고하여 DevMatch 플랫폼의 수강 신청 폼을 개선합니다.

F-Lab 지원서의 핵심 컨셉:
> "시작이 반! 깊이 있는 개발자가 되기 위한 첫걸음.
> 작성에 들인 노력만큼 잘 맞는 멘토님을 만나실 수 있을 겁니다 :)"

즉, **지원서 내용을 기반으로 멘토 매칭의 정확도를 높이는 것**이 핵심 목적입니다.

### 현재 vs F-Lab vs 개선안 비교

| 구분 | 현재 DevMatch (`/apply`) | F-Lab 실제 폼 | 개선안 |
|------|------------------------|---------------|--------|
| 폼 구성 | 3단계 (기본정보→옵션→결제) | 단일 페이지 질문형 | Q&A 스타일 단일 페이지 |
| 개발 배경 | 레벨 3단계만 선택 | 언어/플랫폼/전공/경력/프로젝트 수 등 상세 | F-Lab과 동일하게 상세 수집 |
| 학습 가용 시간 | 없음 | 평일/주말 각각 수집 | F-Lab과 동일하게 수집 |
| 성향/동기 | 목표 커리어 한 줄 | 취업/이직/성장 목표 + 성격 + 자기소개 | F-Lab과 동일 |
| 결제 시점 | 신청과 동시 결제 | 지원서 제출 → 검토 후 결제 | 분리 (지원서 먼저, 결제 나중) |
| 약관 | 없음 | [필수] 서비스 이용 약관 | 동일 |

## 2. 전체 프로세스 흐름

```
[지원서 작성] → [제출 완료 (검토중)] → [합격발표] → [결제] → [멘토 매칭]
```

### 상태 흐름도

| 상태 | 설명 | 다음 상태 |
|------|------|-----------|
| `DRAFT` | 지원서 작성 중 (임시 저장) | `SUBMITTED` |
| `SUBMITTED` | 제출 완료, 검토 대기 | `ACCEPTED` / `REJECTED` |
| `ACCEPTED` | 합격 (결제 대기) | `PAYMENT_COMPLETED` |
| `REJECTED` | 불합격 (사유 안내) | - |
| `PAYMENT_COMPLETED` | 결제 완료 → 멘토 매칭 진행 | - |

## 7. 데이터 모델 (Backend)

### Application Entity

```
Application {
  id: Long (PK)
  userId: Long (FK → User)
  courseId: String                   // 코스 식별자 (예: "java-backend")
  startMonth: String                // "2026-07"
  status: Enum (DRAFT | SUBMITTED | ACCEPTED | REJECTED | PAYMENT_COMPLETED)

  // 섹션 A: 개발 배경
  languages: List<String>           // ["Java", "Javascript", ...]
  platforms: List<String>           // ["웹", "안드로이드", ...]
  isCsMajor: Boolean                // true=전공자, false=비전공자
  learningPaths: List<String>       // ["대학교", "독학", ...]
  careerYears: String               // "경력 없음", "3년차 미만" 등

  // 섹션 B: 프로젝트 경험
  githubUrl: String (nullable)      // Github 주소
  projectCount: String              // "없음", "1개", ... "6개 이상"
  projectDescription: String        // 개발 경험 서술

  // 섹션 C: 학습 계획 및 성향
  weekdayStudyHours: String         // "2시간 미만", "2~4시간" 등
  weekendStudyHours: String         // "2시간 미만", "2~4시간" 등
  goal: String                      // "취업", "이직", "성장"
  personality: String               // "외향적", "내향적"
  selfIntroduction: String          // 자기소개

  // 섹션 D: 기타
  referralSources: List<String>     // ["검색", "지인추천", ...]
  referralCode: String (nullable)   // 추천코드
  termsAgreed: Boolean              // 약관 동의

  // 메타
  submittedAt: LocalDateTime
  createdAt: LocalDateTime
  updatedAt: LocalDateTime
}
```

### API 엔드포인트

| API | Method | 설명 |
|-----|--------|------|
| `/api/applications` | POST | 지원서 생성 (초안 저장) |
| `/api/applications/{id}` | GET | 지원서 상세 조회 |
| `/api/applications/{id}` | PUT | 지원서 수정 (임시 저장) |
| `/api/applications/{id}/submit` | POST | 지원서 최종 제출 |
| `/api/applications/me` | GET | 내 지원서 목록 |
| `/api/admin/applications` | GET | [관리자] 지원서 목록/필터링 |
| `/api/admin/applications/{id}/review` | PUT | [관리자] 합격/불합격 처리 |

## 8. 기존 `/apply` 페이지와의 관계

현재 `/apply` 페이지는 **기본정보 선택 + 결제**가 합쳐진 형태입니다.
개선 후에는 두 흐름을 분리합니다:

| 페이지 | 역할 | 진입 시점 |
|--------|------|-----------|
| `/apply` (신규) | 지원서 작성 폼 (이 문서의 내용) | 코스 상세 페이지에서 "지원하기" 클릭 |
| `/apply/payment` (기존 결제 로직 이동) | 합격 후 결제 | 합격 알림 메일의 결제 링크 클릭 |

기존 `/apply`의 **레벨 선택, 카테고리 선택, 기술 스택 칩** UI 컴포넌트는
새 지원서 폼의 `multi-chip`, `radio` 스타일에 재활용합니다.

## 11. 팀 논의 필요 사항

1. **검토 프로세스:** MVP에서 관리자 검토를 포함할지, 자동 승인으로 처리할지
2. **결제 시점 분리:** F-Lab처럼 합격 후 결제로 변경할지, 현재처럼 즉시 결제를 유지할지
3. **Github 주소:** 필수로 할지 선택으로 할지 (입문자 배려 vs 멘토 매칭 정보)
4. **질문 커스터마이징:** 코스별로 질문이 달라져야 하는지 (예: Frontend 코스는 언어 선택지가 다름)
5. **자기소개 최소 글자 수:** F-Lab은 제한 없음, 최소 글자 수(예: 50자)를 둘지 여부
