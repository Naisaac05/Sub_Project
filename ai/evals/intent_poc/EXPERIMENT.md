# 의도분류 PoC 실험 보고서

> AI 학습 복습 서비스에서 **학습자가 꼬리질문을 했을 때 그 "의도"를 자동으로 분류**하는
> 방법을 6가지로 비교 실험한 기록. 규칙 / 학습형 / 임베딩 / 로컬 LLM을 같은 데이터로 측정했다.

---

## 0. 한 장 요약 (TL;DR)

| 분류기 | 한 줄 설명 | holdout 정확도(진짜 실력) | 호출당 지연 | 사람이 하는 일 |
|---|---|---:|---:|---|
| **embed (bge-m3)** | 의미 임베딩 최근접 | **99.2%** 🥇 | ~2.3초(CPU) | 예시만 추가 |
| **ml (학습형)** | 문자 n-gram + 회귀 | **95.8%** 🥈 | **~0초** | 데이터만 추가 |
| LLM qwen3:4b | 로컬 LLM | 93.3% | 4.4초 | 프롬프트 작성 |
| **rule10 (규칙)** | 키워드 if-else | 91.7% | ~0초 | 규칙 직접 작성 |
| LLM exaone3.5:2.4b | 로컬 LLM | 89.2% | 3.8초 | 프롬프트 작성 |
| LLM qwen2.5:3b | 로컬 LLM | 85.0% | ~수초 | 프롬프트 작성 |
| (참고) 기존 4-intent 규칙 | 옛 규칙 | 35.8% | ~0초 | — |

**결론 두 줄:**
1. **사람이 규칙을 손으로 안 써도 된다.** 데이터로 학습한 `ml`(95.8%)과 예시로 분류하는 `embed`(99.2%)가 손으로 짠 규칙(91.7%)보다 **더 정확**하다.
2. **이 태스크에 LLM은 필요 없다.** LLM은 정확도도 중간이고 가장 느리다(4초). 정답은 `ml`(빠름) + `embed`(정밀)의 조합.

## 1. 배경 — 무엇을 푸는 실험인가

### 🔰 의도분류(intent classification)가 뭔가요?

학습자가 질문을 던지면, 시스템은 먼저 **"이 사람이 원하는 게 뭐지?"**를 알아야 알맞게 답할 수 있습니다.

- "트랜잭션이 뭐야?" → **개념 정의**를 원함
- "세션이랑 JWT 차이가 뭐야?" → **두 개념 비교**를 원함
- "점심 뭐 먹지?" → 복습과 **무관한 잡담**

질문을 **미리 정해둔 10개 "서랍(라벨)" 중 하나에 넣는 것**이 의도분류입니다.

### 10개 서랍(의도)

ANSWER_REASON(정답 이유) · WRONG_ANSWER_REASON(내 오답 이유) · CONCEPT_DEFINITION(개념 정의) ·
COMPARISON(비교) · EXAMPLE_REQUEST(예시) · PRACTICAL_USAGE(실무) · DEBUG_OR_ERROR(에러) ·
FOLLOW_UP(되묻기) · OFF_TOPIC(잡담) · UNKNOWN(불명)

## 2. 먼저 알아둘 용어 (입문자용)

> **분류 방식 4가지 — 핵심 차이는 "사람이 무엇을 하느냐"**
> - **규칙기반(rule)**: 사람이 "이 단어 있으면 이 서랍" if-else를 **직접 작성**. 빠르고 투명하지만 손이 많이 감.
> - **학습형(ml)**: 사람은 규칙을 안 쓰고 **정답 달린 예시(데이터)만** 준다. 모델이 패턴을 스스로 학습.
> - **임베딩(embed)**: 문장을 "의미 좌표"로 바꿔 비슷한 예시에 붙인다. 사람은 **예시 문장만 추가**.
> - **LLM**: 큰 언어모델이 문장을 "읽고 이해해서" 고른다. 유연하지만 느리고 답이 흔들림.

> **문자 n-gram(char n-gram)이 왜 오타에 강한가? (ml의 핵심)**
> 문장을 단어가 아니라 **2~5글자 조각**으로 잘라 본다. '정답'이 오타로 '저답'이 돼도 '답', '저답' 같은
> 조각 대부분이 살아남아 알아본다. 그래서 오타에 강함.

> **임베딩(embedding)이 뭔가? (embed의 핵심)**
> 문장을 **1024개 숫자(좌표)**로 바꿔 "의미 공간의 한 점"으로 만든다. 뜻이 비슷하면 가까운 점이 된다.
> 각 의도의 예시들 **평균점(중심)**에 가장 가까운 의도로 분류. 글자가 아니라 의미로 보니 오타·말바꿈에 강하다.

> **dev / holdout 분리**
> dev = 모델을 만들/학습할 때 보는 "연습 문제집"(120개). holdout = 끝나고 딱 한 번 채점하는 "실전 모의고사"(120개).
> 👉 만들 때 holdout을 보면 컨닝 → **holdout 점수가 진짜 실력.** (특히 ml/embed는 dev로 학습하므로 dev 점수는 '외운 것'이라 무의미.)

> **정확도 · 정밀도 · 재현율 · F1 · macro-F1**
> 정확도=전체 중 맞춘 비율 · 정밀도=X라 예측한 것 중 진짜 X 비율 · 재현율=진짜 X 중 맞춘 비율 ·
> F1=둘의 균형 · macro-F1=10개 서랍 F1을 똑같은 비중으로 평균.

## 3. 실험 데이터

- **시드 30개**(의도 10 × 3) — 사람이 손으로 쓰고 라벨링한 유일한 부분.

- 변형 축: `seed`(원본 30) · `typo`(오타 80) · `filler`(군말 30) · `paraphrase`(말바꿈 100).

## 4. 실험 방법

**단일변인 통제.** 데이터(240)·평가 방식·채점표를 고정하고 **분류기만 교체**한다.
ml/embed는 **dev(120)로만 학습/적합**하고 holdout(120)으로 평가한다(holdout은 학습에 절대 안 씀).

### 골든셋을 분류기마다 어떻게 쓰나 (학습 vs 시험)

> 🔰 dev = **연습문제집(답 보며 공부)**, holdout = **처음 보는 실전 모의고사**.

| 분류기 | dev 120 | holdout 120 | 비교에 쓰는 점수 |
|---|---|---|---|
| rule10 / llm | 학습 안 함(설계 참고만) | 평가 | **holdout** |
| **ml / embed** | **여기서 학습/적합** | 평가(처음 보는 데이터) | **holdout** |

- 학습형(ml/embed)의 **dev 점수 100%는 "외운 것"이라 무의미** → holdout만 진짜 성능.
- 6종 전부 **같은 holdout 120개**로 채점 → 공정한 비교.

```
같은 dataset.jsonl ─> [rule10] [ml] [embed] [llm×3] ─> 같은 채점
```

## 5. 실험 결과

### 5-1. 총괄 비교

| 분류기 | 전체 | dev | **holdout(진짜)** | macro-F1 | 오타(typo) | 호출당 지연 |
|---|---:|---:|---:|---:|---:|---:|
| 기존 4-intent 규칙 | 37.5% | 39.2% | 35.8% | 0.312 | 36.2% | ~0초 |
| **rule10 (규칙)** | 95.0% | 98.3% | 91.7% | 0.951 | 88.8% | ~0초 |
| **ml (학습형)** | 97.9% | 100%* | **95.8%** | 0.979 | **100%** | **~0초** |
| **embed (bge-m3)** | 99.6% | 100%* | **99.2%** | 0.996 | 98.8% | ~2.3초(CPU) |
| LLM qwen2.5:3b | 87.1% | 89.2% | 85.0% | 0.869 | 85.0% | ~수초 |
| LLM exaone3.5:2.4b | 91.2% | 93.3% | 89.2% | 0.906 | 88.8% | 3.8초 |
| LLM qwen3:4b-q4_K_M | 92.9% | 92.5% | 93.3% | 0.928 | 92.5% | 4.4초 |

`*` ml/embed의 dev 100%는 **학습 데이터를 외운 값**이라 의미 없음 → holdout만 진짜 성능.

### 5-2. 오타(typo) 강건성 — 규칙의 약점을 누가 메우나

| 분류기 | 오타 정확도 |
|---|---:|
| rule10 (규칙) | 88.8% ← 규칙은 오타에서 무너짐 |
| LLM (최고 qwen3:4b) | 92.5% |
| **ml (문자 n-gram)** | **100%** |
| **embed (bge-m3)** | **98.8%** |

> 규칙이 오타에 약했던 이유(글자가 깨져 키워드를 못 잡음)를, **문자 n-gram(ml)과 의미 임베딩(embed)이 거의 완벽히 해결**했다. 이게 "표현 변형은 규칙 추가가 아니라 학습/임베딩으로 푼다"의 실측 증거.

### 5-3. 클래스별 성적 (rule10 예시 — 좋은 분류기의 모습)

10개 서랍 전부 F1 0.85~1.00. (ml·embed는 이보다 같거나 높았다.) 기존 4-intent 규칙은 4개 서랍을 아예 못 만들어 재현율 0이었던 것과 대비된다.

## 6. 해석 (입문자용)

### ① "규칙을 사람이 영원히 손으로 추가해야 하나?" → 아니다 (이번 핵심 답)

| 방식 | 사람이 하는 일 | holdout | 손 부담 |
|---|---|---:|---|
| rule10 | 규칙 문장 직접 작성 | 91.7% | 무거움 |
| ml | **데이터(예시+정답)만 추가** | 95.8% | 가벼움 |
| embed | **예시 문장만 추가** | 99.2% | 가장 가벼움 |

뒤로 갈수록 사람은 "**규칙 작성자**"가 아니라 "**데이터 큐레이터**"가 되고, 손 부담은 줄어드는데 정확도는 **오히려 올라간다.** 즉 규칙을 손으로 메우는 건 유일한 길이 아니고, 보통 가장 나쁜 길이다.

### ② 정확도는 비슷한 것끼리 묶인다 (통계 주의)

holdout은 120문제 시험이라 점수에 **±약 5점 운**이 낍니다. rule10(91.7)·qwen3(93.3)·ml(95.8)은 사실상 한 묶음. 다만 **embed는 120개 중 1개만 틀려(99.2%)** 다른 것들과 분명히 구분됩니다.

### ③ 그럼 무엇으로 고르나? → 정확도 + 속도 + 손 부담

- **ml**: holdout 95.8%, **지연 0초**, 오타 100%, 규칙 안 씀 → **종합 가성비 1위**(hot-path 적합).
- **embed**: holdout 99.2%(최고)지만 CPU에서 **~2.3초/건**(GPU면 빨라짐) → 정밀이 필요한 곳에 선택적.
- **LLM**: 정확도 중간 + 가장 느림(4초) → 이 태스크에선 쓸 이유가 약함.

## 7. 결론 & 다음 단계

### 권장 구조 — 계단식(cascade), LLM은 거의 안 씀

```
1단계  ml(문자 n-gram)   ── 약 95% 즉답 (0초, 오타에 강함, 규칙 안 씀)
   └ 저확신만 ↓
2단계  embed(bge-m3)     ── 의미로 정밀 판정 (99%, CPU면 ~2초라 소수에만)
   └ 그래도 애매하면 ↓
3단계  로컬 LLM           ── 아주 드문 폴백 + 구조화 추출 (개념/범위/근거 동시)
```

> 투명성·0의존성이 최우선이면 1단계를 `rule10`(규칙)으로 둬도 됨 — 단 정확도·오타 강건성은 ml이 우위.

### 모델/방식 선택 결론

- **사람이 규칙을 손으로 다 메울 필요 없음** — `ml`/`embed`가 데이터/예시 추가만으로 더 잘함.
- **LLM은 이 분류 태스크의 정답이 아님** — 정확도·지연 모두 열세.
- "작은 모델 vs 큰 모델"은 **태스크별로 측정해서** 결정 (생성 우승 ≠ 분류 우승).

### 다음 단계

1. `ml`(또는 `rule10`)을 참조로 **Phase 1 의도 추출기**를 `app/`에 구현.
2. 표현 변형/정밀이 필요하면 **embed(bge-m3)**를 2단계로 (설계 문서의 `AI_REVIEW_VECTOR_ENABLED` 활성화와 연결).
3. `dataset.jsonl`을 pytest 회귀 테스트로 박아 성능 하락을 자동 감지.
4. 운영 루프: 저확신/UNKNOWN 케이스 수집 → 라벨 → **데이터에 추가**(규칙 작성 아님) → 재측정.

## 부록 D. embed(임베딩) 자세히 — 입문자용

### 1) 컴퓨터는 글자를 못 읽는다 → 문장을 "좌표"로 바꾼다

임베딩 모델(bge-m3)은 문장 하나를 **1024개의 숫자 목록(= 벡터 = 좌표)**으로 바꾼다.
비유하면 **1024차원 공간(지도) 위의 한 점**. 핵심 마법은 **뜻이 비슷하면 가까운 점**이 되도록
모델이 미리 학습돼 있다는 것이다.

```
"트랜잭션이 뭐야?"    → 점 A
"트랜잭션 뜻 알려줘"  → 점 A 바로 옆  (글자는 달라도 뜻이 같아 가깝다)
"점심 뭐 먹지?"       → 멀리 떨어진 점
```

글자가 아니라 **의미**로 위치가 정해지므로, 오타('저답')나 말바꿈도 원래 뜻 근처에 찍힌다.
→ 이게 embed가 오타·말바꿈에 강한 이유.

### 2) 분류 = "가장 가까운 동네 찾기" (최근접 중심, nearest centroid)

- **준비(학습):** 의도(서랍)마다 예시들을 점으로 찍고, 그 점들의 **평균 위치 = 그 의도의 중심점**.
  → COMPARISON 동네의 중심, OFF_TOPIC 동네의 중심 … 이렇게 **10개 동네의 중심점**을 만든다.
- **분류:** 새 질문도 점으로 찍어, **10개 중심 중 가장 가까운 동네**의 의도로 판정.
  "가깝다"는 **코사인 유사도**(두 점의 방향이 얼마나 같은가, 1에 가까울수록 비슷)로 잰다.

```
새 질문 점  ──코사인 거리 비교──>  [DEF중심] [CMP중심] ... [OFF중심]
                                      └─ 가장 가까운 동네의 의도로 분류
```

### 3) 우리가 실제로 한 절차

1. dev 120개 질문을 bge-m3로 임베딩 → 의도별 **중심점 10개** 계산 (학습)
2. holdout 120개를 각각 임베딩 → 가장 가까운 중심의 의도로 분류 → 채점
3. 결과: 120개 중 **119개 정답 = 99.2%**

### 4) 강점과 약점

- 👍 의미 기반이라 오타·말바꿈에 강함(holdout 99.2%, 오타 98.8%). 규칙을 안 쓰고 **예시 문장만 추가**하면 새 표현을 커버.
- 👎 임베딩 1회가 CPU에선 ~2.3초(GPU면 훨씬 빠름). bge-m3 모델 1.2GB 필요. "왜 그렇게 분류했나"가 규칙보다 덜 투명.
- 개선 여지: 한 의도 안에서 표현이 매우 다양하면 '중심 1개'로 부족할 수 있음 → kNN(가까운 예시 여러 개 투표) 등으로 확장 가능.
