## 1. 목표

학습자가 코드리뷰/개념 질문을 했을 때 **빠르고(체감 5~10초) 정확한** 답변을 내놓는다. 단, "특정 버그를 고친다"가 아니라 **구조를 제대로 잡는 것**이 목적이다. 동시에 이 프로젝트는 **학생 포트폴리오/면접 자료**이므로, 설계가 면접에서 설명 가능한 엔지니어링 판단을 담아야 한다.

## 2. 제약 조건 (모든 결정의 근거)

| 제약 | 값 | 영향 |
|---|---|---|
| 하드웨어 | 학생 노트북 16GB, GPU 없음 | 모델 크기 ≤ ~4B(양자화) |
| 비용 | 0원 | 라이브 경로에 유료 API 금지 |
| 네트워크 | 인터넷 의존 없음(셀링포인트) | 라이브는 로컬 전용 |
| 모델 속도 | 로컬 모델 완전한 답 1개 ≈ 14~22초 | "빠름"은 곧 "모델을 덜 부르기" |
| 지연 목표 | **첫 문장 5~10초**(스트리밍, 완성본 아님) | 스트리밍 필수 |

> 핵심 통찰: 위 제약에서 "빠름"의 정체는 대부분 **"느린 로컬 모델을 얼마나 자주 안 부르느냐"**이다.

## 3. 핵심 전략 (한 문장)

> **실시간으로 깊은 출력 검증이 불가능하므로, 정확도는 "앞단(의도 분류 → 정밀 검색 → 좋은 코퍼스)"에서 *예방*으로 확보하고, 깊은 검증은 *비동기·오프라인*으로 미뤄 학습 루프에 반영한다. 모르면 솔직히 폴백한다.**

## 4. 전체 구조 — 안 C: 계층형 하이브리드

채택 배경: 안 A(모델 최소화)는 빠르나 새 질문 약함, 안 B(모델 신뢰+검증)는 정확하나 느리고 정체성 충돌. **안 C는 둘의 균형 + 구조 통합 효과.**

```
[느린 고리 — 주기적·오프라인]              [빠른 고리 — 매 요청·라이브]

쌓인 Q&A (후보 큐)                          질문 들어옴
   │                                          │
   ▼                                          ▼
Claude/GPT 검증 (환각·위험 판정)            의도분류 (규칙)
   │                                          │
   ▼                                          ▼
사람 승인 → 지식카드 저장 ───────────►   좋은 코퍼스에서 정밀 검색
   (= 코퍼스가 학습)                          │
                                              ▼
                                    Tier0 카드 즉답  또는  로컬모델 근거기반 생성(스트리밍)
                                              │
                                              ▼
                                    규칙 검증 → 신뢰도 게이트 → 폴백 → 캐시/후보적재
```

## 5. 컴포넌트별 책임 ("부품 하나 = 책임 하나")

| 부품 | 책임 (딱 하나) | 상태 |
|---|---|---|
| `pipeline.py` *(신규)* | 단계 순서를 **한 곳에만** 정의 → 3개 실행기가 공유 | 신규 |
| `retrieve_context` | 오타교정 + 의도분류(규칙) + RAG 검색(게이트) | 기존 유지 |
| `route_tier` *(신규)* | Tier0 / Tier1 **결정만** | 신규 |
| `instant_answer` (Tier0) | 캐시/템플릿/카드에서 **꺼내오기만** | 기존 통합 |
| `generate` (Tier1) | 모델 호출 + 후처리 **생성만**(310줄→축소) | 기존 축소 |
| `validate` | 규칙 검증(한글률·금지표현·관련성) | 기존 유지 |
| `confidence_gate` | 신뢰도 점수 | 기존 유지 |
| `fallback` | 낮으면 안전한 답으로 교체 | 기존 유지 |
| `persist` | 캐시 저장 + 후보 적재(학습루프) | 기존 유지 |

### 구조 개선 (현재 문제 해결)
- **파이프라인 3중 중복 제거**: 지금 동일 순서가 `graph.py`(LangGraph)·`runner.py:415`(순차)·`runner.py:250`(스트리밍)에 손으로 3번 적혀 있고 이미 드리프트됨. → `pipeline.py` 1곳으로 통합, 스트리밍은 "생성 단계에서 토큰을 흘려보낸다"는 차이만.
- **`generate_answer_node`(약 310줄) 축소**: 빠른답·캐시 판단을 `route_tier`/`instant_answer`로 이관, 생성만 남김.

## 6. 데이터 흐름

### 라이브 (빠른 고리, 스트리밍 기본 ON)
1. 질문 → `retrieve_context`(오타교정 → 의도분류 → 검색, 게이트 5.0)
2. `route_tier`:
   - **Tier0**: 캐시/정적 템플릿/승인 카드 매칭 → 즉답(0.x초)
   - **Tier1**: 로컬 모델 + RAG, **스트리밍**(첫 문장 2~3초), 후처리(strip_markdown, ends_with_question)
3. `validate`(규칙) → `confidence_gate` → 낮으면 `fallback`
4. `persist`(캐시 저장 + 후보 적재)

### 오프라인/비동기 (느린 고리, 학습 루프)
- **비동기(라이브 직후, 백그라운드)**: 로컬 grounding 검사 → 환각 의심 시 캐시 안 함 + "사람 검토" 큐 플래그.
- **오프라인(주기적, 따로)**: Claude/GPT가 표본을 정밀 검증 → ① 전체 품질 측정(golden, `--real`) ② 카드 승급 1차 선별 ③ 로컬 검증기 후보 채점.

## 7. 검증 전략 — 실시간 규칙 + 미룬 깊은 검증

| | 라이브(실시간) | 비동기/오프라인 |
|---|---|---|
| 수단 | 규칙(한글률·금지표현·관련성) + 신뢰도 게이트 + 폴백 | 로컬 grounding(비동기) / Claude(오프라인) |
| 잡는 것 | 형식·안전·자신감 | 사실 오류·환각 |
| 비용/지연 | 0원, 0지연 | 표본만큼(제한적) |

**원칙:**
- 라이브 LLM 검증은 **하지 않는다**(속도·로컬 4b 검증 무능·정체성 충돌). 근거: 자체 실험 결론 "로컬 작은 모델은 구조화 평결 불가".
- 깊은 검증은 미루되, **안 B(위험질문 동기 검증) 뼈대는 코드에 남겨** 나중에 스위치로 켤 수 있게 한다.
- 검증의 수혜자는 "현재 사용자"가 아니라 **"다음 사용자 + 지식 베이스"**. → **모델이 학습하는 게 아니라 시스템(코퍼스)이 학습한다.**

## 8. 의도 분류 = 정확도의 키스톤

깊은 출력 검증을 미루면 정확도 무게중심이 **앞단(라우팅)**으로 이동한다. 의도가 ① RAG 정책 ② 프롬프트 ③ Tier0 진입 여부를 모두 결정.

- **규칙(정규식/키워드) 유지** — 의도는 *중요하지만 쉽고(좁은 분류) 뜨거운(매 요청·맨앞)* 작업 → **규칙에 최적, API에 최악**(핫패스 지연·비결정성·전트래픽 비용).
- **불확실(confidence 낮음)** → 억지 템플릿 금지, 일반 경로/폴백 또는 **되물음(꼬리질문)** — 약점을 제품 특성(대화형 튜터)으로 전환.
- **Claude는 오프라인에서 의도 규칙 개선에만**: 로그를 Claude로 분류 → 규칙과 불일치 발굴 → 새 키워드·골든 케이스 추가. (지능을 값싼 규칙으로 "컴파일")

## 9. RAG 품질 + 정밀 검색 = 정확도 엔진

작은 모델은 "무엇을 보여주느냐"가 답 품질을 거의 결정(garbage in, garbage out). 두 개의 별개 레버:
- **A. 좋은 정보(코퍼스)** — 지식카드 품질·커버리지(재료).
- **B. 정밀 검색** — 이 질문에 맞는 카드 꺼내기(고르기). 쿼리 구성·임베딩·리랭커·**게이트(약하면 빼기)**.

뉘앙스: ① 약한 매치는 차라리 안 넣음(정밀>재현) ② 모든 질문이 RAG를 쓰진 않음(follow-up은 검색 건너뜀) ③ 코퍼스 개선은 Tier0·Tier1 양쪽 동시 향상(더블윈). 측정: golden `expected_concepts`. 작업: `retriever.py`의 `TOKEN_ALIASES`에 음차/오타 별칭 추가.

## 10. 콘텐츠 소스 — F-Lab Java 기술면접 자료집 채택

`[F-Lab] Java 기술면접 자료집`(12p, Top 20문항)을 **콘텐츠/평가 시드로 공식 채택**한다. 문서 철학(정답 미제공·검색키워드·꼬리질문)이 본 설계와 동일.

| 문서 요소 | 시스템 반영 |
|---|---|
| 질문 (equals/hashCode, ArrayList 내부, Blocking IO…) | 지식카드/개념질문 |
| 검색 키워드 | RAG 코퍼스 / `TOKEN_ALIASES` / 의도 topic |
| 잘못 답변할 수 있는 케이스 | **결정론적 금지표현 규칙** + `wrong_answer_explanation` 의도 |
| 꼬리질문 체인 | follow-up 시스템 학습·평가 정답지 |

**자산:** ① 20문항 → 검증된 Tier0 카드(질문 1 equals/hashCode는 이미 [concepts/java/equals.md](../../../ai/app/knowledge/concepts/java/equals.md) 존재) ② 골든 평가셋 시드(질문+expected_concepts+forbidden) ③ 꼬리질문 체인 데이터 ④ 검색 어휘.

**핵심 이점:** 정형화된 면접 도메인은 정답·오답이 고정 → "흔한 오답"을 **금지표현 규칙으로 인코딩**하면 라이브에서 LLM 없이도 사실 오류를 잡는 *예외적* 케이스가 된다. ("품질 확인 없이 정확?" 우려를 이 도메인에선 상당 부분 해소.)

## 11. 모델 선택

실제 코드 현황([client.py:19-20](../../../ai/app/ollama/client.py)): `DEFAULT_MODEL`(라이브 기본) = **`qwen3:1.7b`**, `FALLBACK_MODEL` = **`exaone3.5:2.4b`**. 즉 좋은 모델(exaone)이 현재 *폴백으로만* 쓰이고, 최악 모델(qwen3:1.7b)이 기본이다 → **역할을 바꿔야 한다.**

| 역할 | 목표 모델 | 근거 |
|---|---|---|
| 생성기(라이브) | **exaone3.5:2.4b** ← `DEFAULT_MODEL` 승격 | 자체 블라인드 평가: 의미 0.47, 한국어·소크라테스 부합. 현 기본 qwen3:1.7b는 0.17(최악) |
| 폴백 | qwen3:4b-q4_K_M (정확도 우선) | 주모델 승격 후 폴백 재배치. 현재 폴백 exaone은 주모델로 이동 |
| 검증기(오프라인) | Claude/GPT | 구조화 평결 가능, 표본만 사용 |
| 검증기(비동기 후보) | 측정 후 결정(7B급 후보) | 좁은 grounding 작업은 별도 측정 필요 |

**모델 무관 설계**: 파이프라인은 모델 교체가 설정 한 줄이 되도록 추상화. (면접: "예산/요구가 다르면 프론티어를 라이브에 넣는다 — 설정 한 줄")

## 12. 에러 처리 / 폴백

- 각 단계 실패 → `error_state` → `fallback`(안전한 템플릿). 완전 실패 → `dead_end` → 솔직한 "확실치 않다" 답.
- 생성 예외 → 폴백 모델 1회 재시도 → 실패 시 한국어 템플릿.
- 스트리밍 중 비검증 초안은 도로 빼지 않음(재생성은 비동기로). 사용자 경험 일관성 유지.

## 13. 테스트 / 측정

- **결정론(시스템 배관)**: golden `golden_dataset.jsonl` — 의도/검색/컨텍스트/금지표현 회귀.
- **모델 품질(`--real`)**: 실 Ollama 호출 평가, 모델 회귀 감시.
- **지연**: 첫 토큰 시간(TTFT) 측정 — 5~10초 목표 검증.
- **검증기 채점**: Claude 정답지로 로컬 검증기 후보 비교(오프라인).

## 14. 단계적 구현 순서 (우선순위)

1. **`pipeline.py` 통합** — 3중 중복 제거(가장 큰 구조 레버). 스트리밍 기본 ON.
2. **모델 교체** — `qwen3:1.7b → exaone3.5:2.4b` + strip_markdown/ends_with_question를 `nodes.py`로.
3. **F-Lab 20문항 → 지식카드 + 골든 행** (콘텐츠 커버리지).
4. **`route_tier`/`instant_answer` 분리** — generate 축소.
5. **비동기 grounding** — 학습 루프 연결.
6. **TOKEN_ALIASES 음차/오타 보강** — RAG 정밀도.
7. (선택) 위험질문 동기 검증 스위치 + 로컬 검증기 측정.

## 15. 면접 포인트 (포트폴리오)

> **Q. "왜 Claude/GPT를 안 썼나요?"** (이 프로젝트의 핵심 질문)
> A. (재구성) "씁니다 — 라이브가 아니라 **오프라인 평가자·정답지 심판**으로요. '로컬이냐 프론티어냐'가 아니라 '각각을 어디에 둘 것이냐'였습니다."
> ① (제약을 의도로) "전제가 '노트북 한 대, 0원, 인터넷·데이터 유출 없이 신뢰할 코드리뷰 튜터'였습니다 — 비용·프라이버시·오프라인이 중요한 실제 제약."
> ② (작은 모델이 더 어렵다) "GPT 호출은 API 한 줄, 누구나 합니다. 흥미로운 엔지니어링은 *작은* 모델을 신뢰할 만하게 만드는 것 — RAG·의도 라우팅·신뢰도 게이트·사람루프·오프라인 평가."
> ③ (측정) "추측하지 않고 golden·`--real`로 *어디서* 충분/부족한지 수치화해 라우팅했습니다."
> ④ (바꿀 수 있게 설계 — 킬러) "예산·요구가 다르면 프론티어를 라이브에 넣습니다. 파이프라인을 **모델 무관**하게 짜서 바꾸는 건 설정 한 줄입니다."

> **Q. "중요한 의도분류에 왜 큰 모델을 안 썼나?"**
> A. "모델 배치는 *중요도*가 아니라 *난이도×위치*로 정합니다. 의도분류는 **쉽고(좁은 분류)·뜨거운(매 요청 맨앞)** 작업이라 규칙이 최적, API가 최악(핫패스 지연·비결정성·전트래픽 비용). 대신 Claude는 오프라인에서 규칙을 개선하는 데 썼습니다."

> **Q. "작은 모델로 어떻게 정확도를 확보했나?"**
> A. "출력 검증이 아니라 *예방*입니다 — ① 검색 강화 ② 검증 엄격(규칙) ③ 사람-루프. 그리고 **모델이 학습하는 게 아니라 시스템(지식카드)이 학습**합니다. GPU 학습·파인튜닝 없이 시간이 지날수록 똑똑해지는 구조."

> **Q. "품질 확인 없이 답을 내보내도 되나?"**
> A. "라이브엔 규칙 검증·신뢰도 게이트·폴백이 있어 *모를 땐 아는 척 안 합니다*. 깊은 사실 검증은 비동기/오프라인으로 미뤄 다음 사용자와 지식베이스를 보호합니다. 특히 면접 같은 *정형 도메인*은 흔한 오답을 규칙으로 인코딩해 라이브에서도 잡습니다."

> **30초 피치 보강**: "작은 로컬 모델(0원·오프라인)로 신뢰할 코드리뷰를 만들기 위해, 빠름은 스트리밍과 계층형 라우팅으로 체감 확보하고, 정확함은 '앞단(의도·검색·코퍼스) 예방 + 미룬 깊은검증 + 사람루프'로 쌓아 올렸습니다. 프론티어 모델은 싼 곳(오프라인 평가)에만 쓰고 사용자 눈앞은 전부 로컬입니다."
