### ② First Token Latency: 860ms (신규 도입) — ⚠️ 부분 일치 (최신 코드 보정)

> 2026-05-22 현재 **Streaming Response 경로는 구현되어 있습니다.** 다만 860ms 수치는 아직 코드/테스트에서 실측 근거가 확인되지 않았고, Spring streaming 요청 컨텍스트 일부가 stub 값에 의존합니다.

| 포트폴리오 주장 | 코드 실상 | 판정 |
|:---|:---|:---:|
| Streaming Response 도입 | Python `StreamingResponse`, Ollama `stream: true`, Spring `SseEmitter`, Frontend `ReadableStream` 소비 구현 | ✅ |
| First Token Latency 860ms 달성 | 실측 로그/벤치마크 산출 근거는 문서와 코드에서 확인되지 않음 | ⚠️ |
| 프론트엔드 Streaming 표시 | `STREAMING` 상태 전이와 `streamingContent` 렌더링 구현 | ✅ |
| 기존 workflow와 완전 동일 컨텍스트 | `AiReviewStreamingService`의 Python 요청에 `stub question`, `stub correct`, `stub selected`가 남아 있음 | ⚠️ |

**상세 분석**:
- **Python측**: `ai/app/ollama/client.py`에는 기존 동기 `call_ollama()`와 별도로 `call_ollama_stream_async()`가 있으며, Ollama API에 `stream: true`로 요청하고 `httpx.AsyncClient.stream()`으로 line 단위 chunk를 yield합니다. `ai/app/main.py`는 `PYTHON_AI_STREAMING_ENABLED`가 켜지고 payload `stream=true` 또는 `Accept: text/event-stream`인 경우 `StreamingResponse`를 반환합니다.
- **Java측**: `AiReviewController`는 `/sessions/{sessionId}/messages/stream` SSE endpoint를 노출하고, `AiReviewStreamingService`는 `SseEmitter`로 Python SSE를 프록시합니다. 단, 현재 Python 요청 객체를 만들 때 question/correct/selected 값이 stub으로 들어가므로 답변 품질 컨텍스트는 non-streaming 경로와 완전 동등하지 않습니다.
- **Frontend측**: `submitAiReviewAnswerStream()`은 `Accept: text/event-stream`으로 stream endpoint를 호출하고, `page.tsx`는 `STREAMING` 상태와 `streamingContent`를 사용해 chunk를 렌더링합니다.

> **결론**: "Streaming Response 도입"은 현재 코드와 일치합니다. 다만 "First Token Latency 860ms 달성"은 실측 근거가 확인되지 않았으므로, 포트폴리오에서는 "스트리밍 경로를 도입해 first-token 체감 개선 기반을 마련" 정도로 쓰고, 860ms 수치는 실제 측정 후에만 사용해야 합니다.

| P 확률값들 (Before/After) | **실측 텔레메트리에서 산출한 것이 아닌 역산 추정치** | ⚠️ |
