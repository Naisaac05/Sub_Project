# 코스별 실력 테스트 문제 설계안

## 목적

실력 테스트는 사용자가 선택한 멘토링 코스 기준으로 현재 실력을 진단하는 기능이다.

이 테스트의 목적은 추천 코스를 정해주는 것이 아니라, 사용자가 이미 선택한 분야 안에서 다음 정보를 제공하는 것이다.

- 현재 점수
- 현재 상태
- 영역별 강점과 약점
- 멘토가 참고할 수 있는 학습 출발점

## 기본 원칙

- 코스는 사용자가 직접 선택한다.
- 테스트는 선택한 코스 기준으로 출제한다.
- 문제를 입문/중급/고급 세트로 나누지 않는다.
- 각 문제에 난이도와 영역 태그를 부여한다.
- 최종 결과는 총점과 영역별 점수로 보여준다.
- 결과 화면에서 추천 코스는 제공하지 않는다.

## 문제 구성

코스별 테스트는 25문제를 기본으로 한다.

| 유형 | 비율 | 문제 수 | 목적 |
| --- | ---: | ---: | --- |
| 객관식 | 45% | 11문제 | 핵심 개념 이해 확인 |
| 단답/빈칸 | 20% | 5문제 | 용어, 문법, 핵심 키워드 확인 |
| 코드 읽기 | 25% | 6문제 | 코드 흐름, 버그, 출력 결과 이해 |
| 시나리오 판단 | 10% | 3문제 | 실무 상황에서 원인과 해결 방향 판단 |

## 난이도와 배점

문제마다 난이도를 1~5로 둔다.

| 난이도 | 의미 | 권장 배점 |
| --- | --- | ---: |
| 1 | 기본 개념 확인 | 2점 |
| 2 | 간단한 적용 | 3점 |
| 3 | 실무 기본 상황 | 4점 |
| 4 | 복합 상황 판단 | 5점 |
| 5 | 장애, 성능, 설계 판단 | 6점 |

| 난이도 | 문제 수 |
| --- | ---: |

## 시나리오 판단 문제란?

시나리오 판단 문제는 실무 상황을 제시하고, 가장 적절한 원인이나 해결 방향을 고르는 문제다.

예시:

```text
게시글 목록 API가 느려졌습니다.
로그를 확인해보니 게시글 20개를 조회할 때 댓글 조회 쿼리가 게시글마다 한 번씩 추가로 실행됩니다.

가장 먼저 의심할 문제는 무엇인가요?

A. CORS 설정 오류
B. N+1 쿼리 문제
C. 트랜잭션 격리 수준 문제
D. JWT 만료 문제
```

정답: B

## 결과 화면 구성

결과 화면은 다음 정보를 제공한다.

- 선택한 코스명
- 총점
- 현재 상태 문구
- 영역별 점수
- 강점 영역
- 보완 필요 영역
- 틀린 문제와 해설

예시:

```text
Python Backend + AI 실력 테스트 결과

총점: 68점
현재 상태: 프로젝트는 시작할 수 있지만 ORM과 비동기 처리 보강이 필요합니다.

영역별 점수
- Python: 82
- FastAPI/Django: 70
- ORM: 52
- 비동기: 45
- 데이터 처리: 65

강점
- Python 기본 문법
- API 요청/응답 흐름

보완 필요
- ORM 관계 매핑
- 비동기 I/O 처리
- 배포 환경에서의 에러 처리
```

## 멘토 참고 요약

```text

ORM 관계 매핑과 비동기 처리 경험이 부족합니다.
초반 2주는 데이터 모델링과 API 오류 처리 중심으로 진행하는 것이 좋습니다.
```

## 문제 품질 기준

실력 테스트 문제는 합격/불합격을 가르는 시험 문제가 아니라, 멘토링 시작 전 현재 상태를 정확히 진단하기 위한 문제다. 따라서 문제를 만들 때는 아래 기준을 통과해야 한다.

### 1. 측정 목적이 명확해야 한다

문제 하나는 하나의 핵심 개념 또는 하나의 판단 능력을 평가해야 한다.

나쁜 예:

```text
Spring에 대해 설명하시오.
```

좋은 예:

```text
Spring에서 생성자 주입을 사용하는 이유로 가장 적절한 것은?
```

문제마다 아래 정보가 명확해야 한다.

```text
courseKey: java-backend
area: Spring
conceptTags: [Dependency Injection, Bean Lifecycle]
skill: 개념 이해
```

### 2. 정답이 하나로 결정되어야 한다

실무에서는 상황에 따라 답이 달라질 수 있지만, 테스트 문제는 채점 가능해야 한다.

나쁜 예:

```text
좋은 API 설계 방식은 무엇인가요?
```

좋은 예:

```text
리소스 생성에 성공했을 때 가장 적절한 HTTP 상태 코드는?

A. 200
B. 201
C. 400
D. 500
```

### 3. 단순 암기보다 적용과 이해를 우선한다

초반 난이도에서는 용어 확인 문제가 필요하지만, 대부분의 문제는 실제 상황에 개념을 적용할 수 있는지 확인해야 한다.

나쁜 예:

```text
HTTP 404는 무엇인가요?
```

좋은 예:

```text
존재하지 않는 게시글 ID로 상세 조회를 요청했습니다.
가장 적절한 응답 상태 코드는 무엇인가요?
```

### 4. 코스별 영역과 직접 연결되어야 한다

문제는 반드시 선택한 코스의 핵심 영역 중 하나와 연결되어야 한다.

예:

- Java Backend: Java, Spring, JPA, REST API, 트랜잭션
- Frontend: React, Next.js, 상태 관리, 렌더링, CSS/UI
- DevOps: Docker, CI/CD, Linux, AWS, 네트워크

너무 일반적인 CS 문제만 많으면 코스별 테스트의 의미가 약해진다.

### 5. 난이도 기준이 일관되어야 한다

난이도는 출제자의 감이 아니라 기준으로 정한다.

| 난이도 | 기준 |
| --- | --- |
| 1 | 용어와 기본 개념을 알고 있는지 확인 |
| 2 | 간단한 코드나 상황에 개념을 적용할 수 있는지 확인 |
| 3 | 흔한 실무 상황에서 원인을 고를 수 있는지 확인 |
| 4 | 여러 개념이 섞인 상황에서 우선순위를 판단할 수 있는지 확인 |
| 5 | 성능, 장애, 설계 트레이드오프를 판단할 수 있는지 확인 |

JPA 예시:

```text
난이도 1: @Entity의 역할은 무엇인가요?
난이도 2: @ManyToOne 관계의 기본 fetch 전략은 무엇인가요?
난이도 3: 게시글 목록 조회 시 댓글 쿼리가 반복 실행됩니다. 가장 먼저 의심할 문제는?
난이도 4: fetch join을 적용했더니 페이징이 깨졌습니다. 다음 대응으로 적절한 것은?
난이도 5: 대량 트래픽에서 읽기 성능과 데이터 정합성을 함께 고려할 때 조회 모델을 어떻게 분리할 것인가요?
```

### 6. 오답 선택지도 실제로 헷갈릴 만해야 한다

오답이 너무 말이 안 되면 실력 진단에 도움이 되지 않는다.

좋은 오답 예:

```text
A. Fetch Join
B. Batch Size
C. Lazy Loading
D. Cascade Remove
```

### 7. 해설이 학습으로 이어져야 한다

모든 문제에는 정답 해설이 있어야 한다. 객관식 문제는 가능하면 오답 해설도 함께 제공한다.

```text
정답: B

정답 해설:
게시글마다 댓글 조회 쿼리가 추가로 실행되는 상황은 N+1 문제의 전형적인 증상입니다.

오답 해설:
A. CORS는 브라우저 보안 정책 문제라 DB 쿼리 증가와 직접 관련이 없습니다.
C. 트랜잭션 격리 수준은 동시성/정합성 문제와 관련이 큽니다.
D. JWT 만료는 인증 실패와 관련됩니다.
```

### 8. 꼬리질문으로 확장 가능해야 한다

문제는 틀렸을 때 멘토링 대화로 이어질 수 있어야 한다.

```text
문제 개념: JPA N+1

틀린 경우 꼬리질문:
1. N+1 문제가 발생하는 이유를 설명해보세요.
2. Fetch Join과 Batch Size의 차이를 설명해보세요.
3. 페이징이 필요한 목록 API에서 N+1을 줄이는 방법은 무엇인가요?
```

## 꼬리질문 설계 방식

꼬리질문은 틀린 문제를 기반으로 약한 개념을 다시 확인하기 위한 후속 질문이다. 사용자가 점수만 보고 끝나는 것이 아니라, 틀린 개념을 스스로 설명하면서 확실히 정리하게 만드는 것이 목적이다.

흐름:

```text
실력 테스트 응시
→ 틀린 문제 저장
→ 틀린 문제의 conceptTags 추출
→ 약점 개념 저장
→ 꼬리질문 2~3개 제공
→ 사용자 답변 저장
→ 매칭된 멘토가 결과와 답변 확인
```

꼬리질문 유형:

| 유형 | 설명 | 예시 |
| --- | --- | --- |
| 개념 설명형 | 개념을 자기 말로 설명하게 함 | N+1 문제가 무엇인지 설명해보세요. |
| 차이 비교형 | 헷갈리는 개념의 차이를 비교하게 함 | Fetch Join과 Batch Size의 차이는 무엇인가요? |
| 적용 상황형 | 비슷한 상황에 적용하게 함 | 페이징이 있는 목록 API에서는 어떻게 처리할 수 있나요? |
| 원인 분석형 | 증상에서 원인을 찾게 함 | 왜 게시글 수만큼 댓글 쿼리가 추가로 실행될까요? |
| 개선 방향형 | 해결책과 트레이드오프를 말하게 함 | 성능과 페이징을 모두 고려하면 어떤 방식을 선택하겠나요? |

꼬리질문은 정답을 맞히는 시험이 아니라, 멘토가 멘티의 이해 수준을 파악하기 위한 자료로 사용한다.

## 멘토 확인 화면에 보여줄 정보

매칭된 멘토는 멘티의 신청서와 함께 실력 테스트 결과를 확인할 수 있어야 한다.

멘토에게 보여줄 정보:

- 선택한 코스
- 총점
- 현재 상태 문구
- 영역별 점수
- 강점 영역
- 보완 필요 영역
- 약점 태그
- 틀린 문제 목록
- 틀린 문제의 정답과 해설
- 꼬리질문과 멘티 답변
- 멘토 참고 요약

멘토 화면 예시:

```text
Python Backend + AI
총점: 68점

강점
- Python 기본 문법
- API 요청/응답 흐름

보완 필요
- ORM 관계 매핑
- 비동기 I/O
- 배포 환경 에러 처리

꼬리질문 답변 요약
- N+1 문제의 원인은 대략 이해하고 있으나 Fetch Join과 Batch Size 차이는 불명확함
- 비동기 처리를 "속도가 빨라지는 것" 정도로 이해하고 있어 I/O 관점 설명이 필요함
```

## 문제 제작 체크리스트

문제를 등록하기 전에 아래 항목을 확인한다.

```text
[ ] 어떤 코스의 어떤 영역을 평가하는지 명확한가?
[ ] 정답이 하나로 결정되는가?
[ ] 단순 암기보다 적용/이해를 묻는가?
[ ] 난이도가 기준에 맞게 부여됐는가?
[ ] 오답 선택지가 실제로 헷갈릴 만한가?
[ ] 정답 해설이 있는가?
[ ] 오답 해설이 있는가?
[ ] conceptTags가 충분히 구체적인가?
[ ] 틀렸을 때 제공할 꼬리질문 후보가 있는가?
[ ] 멘토가 결과를 보고 수업 방향을 잡을 수 있는가?
```

## 문제 데이터 구조 확장안

추후 꼬리질문과 멘토 확인 기능을 고려하면 문제 데이터는 아래 필드를 포함하는 것이 좋다.

```ts
type SkillTestQuestion = {

  difficulty: 1 | 2 | 3 | 4 | 5;

  explanation: string;
```

## 문제별 예상 풀이 시간

문제 유형과 난이도에 따라 예상 풀이 시간을 둔다. 이 값은 전체 제한 시간 계산, 너무 빠른 제출 감지, 테스트 UX 설계에 사용한다.

| 유형 | 예상 풀이 시간 |
| --- | --- |
| 객관식 | 30~60초 |
| 단답/빈칸 | 45~90초 |
| 코드 읽기 | 90~180초 |
| 시나리오 판단 | 120~240초 |

25문제 기준 전체 테스트 시간은 30~40분을 권장한다.

## 문제 랜덤 출제 규칙

문제는 문제 은행에서 랜덤으로 출제하되, 영역과 난이도 비율은 유지한다.

예: Java Backend 25문제

```text
Java: 5문제
Spring: 5문제
JPA: 6문제
REST API: 4문제
트랜잭션: 5문제
```

각 영역 안에서 난이도 비율을 맞춰 랜덤으로 선택한다. 단, 난이도 4~5 문제는 너무 많이 나오지 않도록 제한한다.

## 같은 영역 문제 중복 방지

같은 영역이라도 같은 개념만 반복해서 나오면 진단 품질이 떨어진다.

중복 방지 규칙:

```text
동일 conceptTag는 한 테스트에서 최대 2문제까지만 출제한다.
난이도 4~5 문제는 동일 conceptTag 중복을 허용하지 않는다.
같은 코드 조각을 기반으로 한 문제는 한 테스트에서 최대 1개만 출제한다.
한 영역의 문제가 전체의 30%를 넘지 않도록 한다.
```

예:

```text
JPA 영역에서 N+1 문제만 4개 출제되면 안 된다.
N+1, 영속성 컨텍스트, 연관관계, 트랜잭션 경계처럼 개념을 분산한다.
```

## 재시험 가능 주기

테스트는 학습 후 재진단 목적이므로 너무 자주 응시하지 않도록 한다.

권장 정책:

```text
같은 코스 테스트는 7일 후 재응시 가능하다.
최신 결과를 기본 결과로 사용한다.
이전 결과는 히스토리로 보관한다.
관리자는 테스트/운영 목적으로 재응시 제한을 초기화할 수 있다.
```

재시험 결과 화면에는 이전 점수와 비교한 변화량을 보여줄 수 있다.

```text
이전 점수: 58점
현재 점수: 71점
변화: +13점
가장 많이 개선된 영역: REST API
아직 보완이 필요한 영역: 트랜잭션
```

## 결과 저장 방식

테스트 결과는 멘토링 진단 자료로 사용되므로 단순 점수만 저장하면 부족하다.

저장해야 할 정보:

- attemptId
- userId
- courseKey
- status: IN_PROGRESS / COMPLETED / ABANDONED / EXPIRED
- startedAt
- submittedAt
- totalScore
- normalizedScore
- areaScores
- questionResults
- wrongQuestionIds
- conceptTags
- weakConceptTags
- followUpAnswers
- elapsedTimeSec

예시:

```ts
type SkillTestAttempt = {
  id: number;
  userId: number;
  courseKey: string;
  status: 'IN_PROGRESS' | 'COMPLETED' | 'ABANDONED' | 'EXPIRED';
  startedAt: string;
  submittedAt?: string;
  totalScore?: number;
  normalizedScore?: number;
  areaScores?: Record<string, number>;
  wrongQuestionIds?: number[];
  weakConceptTags?: string[];
  elapsedTimeSec?: number;
};
```

## 테스트 중 이탈 시 처리

사용자가 브라우저를 닫거나 새로고침해도 진행 상태가 완전히 사라지면 안 된다.

권장 흐름:

```text
테스트 시작 시 attempt 생성
문제 답변 선택 시 서버에 임시 저장
새로고침 또는 재접속 시 IN_PROGRESS attempt 복구
제한 시간이 지나면 자동 제출 또는 EXPIRED 처리
중도 이탈 후 일정 시간이 지나면 ABANDONED 처리
```

이탈 상태도 멘토에게 직접 보여줄 필요는 없지만, 운영자가 확인할 수 있도록 저장한다.

## 진단용 테스트 안내 문구

이 테스트는 합격/불합격 시험이 아니라 멘토링 진단용이다. 따라서 부정행위 방지 문구를 강하게 넣기보다, 솔직하게 풀어야 멘토링 품질이 좋아진다는 방향으로 안내한다.

테스트 시작 전 안내 문구:

```text
이 테스트는 합격/불합격을 판단하는 시험이 아닙니다.
선택한 코스에서 현재 실력을 확인하고,
멘토가 더 정확한 커리큘럼을 설계하기 위한 진단 자료로 사용됩니다.

검색이나 외부 도움 없이 현재 실력 기준으로 풀어주세요.
정확한 진단일수록 멘토링 방향을 더 잘 맞출 수 있습니다.
```

부정행위 방지는 최소한으로만 둔다.

- 문제 복사 방지보다 문제 은행과 랜덤 출제를 우선한다.
- 시험 감시보다 진단 목적 안내를 우선한다.
- 결과는 평가/탈락이 아니라 멘토링 설계 자료로 사용한다.
