# EXAONE Live E2E Quality Evaluation Design

## Goal

Run the real AI review workflow with local EXAONE and compare production-style BGE-M3 RAG answers against forced No-RAG answers for the same questions.

## Scope

The evaluation covers:

- Real `run_review_workflow()` execution.
- Existing free-question intent classification.
- Production-style BGE-M3 retrieval selected through `AI_REVIEW_RAG_RETRIEVER=bge`.
- Forced No-RAG execution with retrieval disabled for evaluation only.
- Real EXAONE `exaone3.5:2.4b` generation.
- Existing workflow validation and semantic judge behavior.
- Rule-based keyword and forbidden-claim checks.
- Answer text, route, retrieved cards, latency, fallback state, quality flags, and observability output.

The evaluation does not change production routing, intent classification, prompts, or answer policy.

## Selected Approach

The evaluator calls `run_review_workflow()` directly. Each dataset row runs twice:

1. `rag`: existing workflow with `AI_REVIEW_RAG_RETRIEVER=bge`.
2. `no_rag_forced`: the same workflow while `app.workflow.nodes.retrieve_context` is patched to return an empty list.

Both modes disable the lightweight/static answer resolver inside the evaluator so every answer reaches live EXAONE generation. Semantic Judge is temporarily disabled inside the evaluator to avoid a second EXAONE request and CPU timeout; production behavior is unchanged. Patching retrieval at the workflow-node dependency boundary keeps intent classification, prompt building, EXAONE generation, and validation identical between modes. This is a generation-forced quality evaluation, not a latency measurement of the default production fast-path mix.

## Dataset

The smoke dataset contains 12 to 15 representative programming questions. It includes:

- Concept definitions.
- Comparisons.
- Practical usage questions.
- Error/debug questions.
- Questions with an approved/generated knowledge card.
- Questions without a strong matching card.

Each row contains:

```json
{
  "id": "rest-api-definition",
  "question": "REST API가 뭐야?",
  "required_keywords": ["HTTP", "자원"],
  "forbidden_claims": ["데이터베이스 인덱스 프로토콜"],
  "human_review_required": true
}
```

## Evaluation Output

For every mode and question, record:

- `answer`
- `model_used`
- `route`
- `retrieved_concept_ids`
- `latency_ms`
- `fallback_used`
- `quality_flags`
- required-keyword pass
- forbidden-claim pass
- semantic judge event fields when present

The JSON report is the machine-readable source of truth. A Markdown report summarizes metrics and includes full answers for human review.

## Metrics

- Required-keyword pass rate.
- Forbidden-claim absent rate.
- Fallback rate.
- Generation route rate.
- EXAONE model-use rate.
- Quality-flag-free rate.
- Latency p50 and p95.
- Per-question RAG versus forced No-RAG differences.

Local Semantic Judge results are advisory. They do not independently determine factual correctness.

## Error Handling

- A failed question is recorded with its exception and evaluation continues.
- If Ollama or EXAONE is unavailable, the evaluator exits non-zero.
- A response that uses a template, cache, or lightweight route remains visible and is not falsely counted as live EXAONE generation.
- Cache is cleared before each mode/question run so answers are freshly generated.

## Verification

```powershell
cd C:\Users\User\Desktop\Sub_Project\ai
.\.venv\Scripts\python.exe -m unittest tests.test_exaone_live_e2e_evaluator -v
.\.venv\Scripts\python.exe scripts/evaluate_exaone_live_e2e.py --limit 12
```

The live evaluation passes execution verification when:

- Both modes execute for every selected question.
- At least one RAG run retrieves a BGE-M3 knowledge card.
- At least one answer is freshly generated by `exaone3.5:2.4b`.
- Reports contain complete answers and comparison metrics.

Quality acceptance is reported separately and is not implied merely by successful execution.
