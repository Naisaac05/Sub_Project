# 운영 AI 후보·RAG 카드 공유 복구 설계

## 목표

운영 Docker 환경에서 승인된 `concepts_v2` RAG 카드가 관리자 후보 화면에 다시 표시되고, RAG 미스 후 Ollama가 생성한 답변이 백엔드 DB의 승인 대기 후보로 저장되도록 한다.

## 아키텍처

EC2 배포 디렉터리의 `./ai/app/knowledge/concepts_v2`를 AI 컨테이너의 기존 카드 경로와 백엔드 컨테이너의 전용 카드 경로에 동시에 bind mount한다. 백엔드에는 `AI_REVIEW_CONCEPTS_V2_PATH`를 전용 경로로 설정한다. 이 방식은 저장소에 포함된 기존 카드를 즉시 제공하며, 관리자 승인으로 생성되는 카드도 EC2 파일시스템에 남긴다.

AI 컨테이너에는 후보 저장 주소를 `http://backend:8080/api/internal/ai-review/candidates/capture`로 명시한다. AI와 백엔드는 동일한 `.env.prod`의 `AI_REVIEW_SERVICE_TOKEN`을 사용하므로 기존 내부 토큰 검증을 유지한다.

## 데이터 흐름

1. 백엔드 후보 목록 API가 RDS의 후보를 조회한다.
2. 백엔드는 공유된 `concepts_v2` 디렉터리에서 승인 카드도 읽어 응답에 합친다.
3. 자유 질문이 승인 카드에 매칭되지 않으면 AI가 Ollama 답변을 생성한다.
4. 후보 생성 조건을 만족하면 AI가 Docker 서비스명 `backend`로 후보를 POST한다.
5. 백엔드가 후보를 RDS에 `PENDING` 상태로 저장하고 관리자 화면에 노출한다.
6. 관리자가 승인하면 백엔드가 공유 디렉터리에 카드를 기록하고 이후 AI와 백엔드가 같은 파일을 사용한다.

## 오류 처리와 관측성

후보 HTTP 저장이 실패하면 URL과 HTTP 상태 또는 예외 종류를 경고 로그로 남기되 서비스 토큰과 전체 답변은 기록하지 않는다. 워크플로 응답에는 기존 `candidate_capture_failed` 품질 플래그를 유지한다.

## 테스트

- compose 렌더링 결과에서 두 서비스의 카드 bind mount와 백엔드 카드 경로를 검증한다.
- compose 렌더링 결과에서 AI 후보 저장 URL이 `backend:8080`인지 검증한다.
- 후보 sink 테스트에서 HTTP 오류와 네트워크 오류가 `False`를 반환하면서 경고 로그를 남기는지 검증한다.
- 기존 AI 후보 저장 및 백엔드 후보 목록 테스트를 실행한다.
- 운영 배포 후 컨테이너 내부 경로, 후보 capture 요청, 관리자 화면 카드 수를 순서대로 확인한다.

## 범위 제외

- 기존 RDS 후보 데이터의 삭제·복원
- 카드 스키마 변경
- Ollama 모델 또는 RAG 검색 알고리즘 변경
- 운영 배포 자체와 데이터 마이그레이션
