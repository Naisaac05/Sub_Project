## 공통

### 접근 제어

- `/admin/payments/**` 경로는 **ADMIN 역할**만. `SecurityConfig.java:49` 공통 가드.

### 라우팅

| 경로 | 화면 |
|------|------|
| `/admin/payments` | 결제 목록 (탭: 전체 / 대기 / 확정 / 취소 / 실패) |
| `/admin/payments/[id]` | 결제 상세 + 환불 액션 |

### API 매핑 (일부 신규 백엔드 작업 필요)

| 동작 | HTTP | 엔드포인트 | body | 현황 |
|------|------|-----------|------|------|
| 목록 | GET | `/api/admin/payments?status={PENDING\|CONFIRMED\|CANCELLED\|FAILED\|ALL}&q=&from=&to=&page=&size=` | — | ❌ 없음, 신규 |
| 상세 | GET | `/api/admin/payments/{id}` | — | ❌ 없음, 신규 |
| 관리자 강제 환불 | POST | `/api/admin/payments/{id}/refund` | `{reason: string, amount?: number}` | ❌ 없음, 신규 |
| 통계 요약 | GET | `/api/admin/payments/summary?from=&to=` | — | ❌ 없음, 신규 (대시보드용) |

> ⚠ **갭 1**: 기존 `PaymentController.java:51` 에 `POST /api/payments/{id}/cancel` 이 있지만 이는 **본인 결제 취소**(userId 검증). 관리자가 다른 사용자의 결제를 강제 환불하려면 **별도 엔드포인트 필요**.

> ⚠ **갭 2**: 부분 환불을 지원하려면 토스 API 연동 방식 재검토 필요. 1차 목업은 **전액 환불만** 지원하는 버전으로 작성. 부분 환불은 백엔드 확인 후 별도 이터레이션.

### 환불 정책

- 환불 가능 상태: `CONFIRMED` 만.
- `PENDING` → 사용자가 결제창 미완료. 환불 불필요, 토스가 자동 만료.
- `CANCELLED` / `FAILED` → 이미 종료 상태, 액션 불가 (버튼 숨김).
- 환불 시 `Matching` 연결이 있으면 함께 처리 필요 (§ 결정 필요 사항).

### 요약 카드 내용

- `총 결제액` = 최근 30일 확정(CONFIRMED) 금액 합
- `확정 건수` = 최근 30일 CONFIRMED 건수
- `환불액` = 최근 30일 CANCELLED 된 결제의 원 금액 합
- `환불률` = 환불 건수 / (확정+환불)

### 기간 필터

- 기본값: 오늘 기준 최근 30일.
- DateRangePicker → 서버에 `from`, `to` 전달.

## 결정 필요 사항

1. **부분 환불 지원 여부** — Phase II에서는 전액만? 토스 API 는 부분 지원함.
2. **환불 시 연관 엔티티 처리**
   - Matching 상태를 `CANCELLED` 로 바꾸기
   - 진행 중인 `MentoringSession` 이 있으면 어떻게?
   - `LmsAssignment` / `AssignmentSubmission` 처리 정책?
   - → 백엔드 서비스 설계가 먼저 필요. 목업은 "함께 취소" 로 명시.
3. **환불 감사 로그** — `Payment.cancelReason` 에 사유가 들어가지만 "누가 처리했는지" 기록 필드 없음. `processedByAdminId` 컬럼 추가 필요.
4. **이메일 알림** — 환불 시 멘티·멘토에게 메일 발송이 의무인지, 선택인지.
