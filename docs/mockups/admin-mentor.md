## 공통

### 접근 제어

- 전체 `/admin/**` 경로는 **ADMIN 역할** 만 접근 가능 (백엔드 [SecurityConfig.java:49](../../backend/src/main/java/com/devmatch/config/SecurityConfig.java) 와 일치).
- 프런트 가드: `app/admin/layout.tsx` 에서 `useAuth()` 로 role 확인 → ADMIN 아니면 **403 페이지 렌더** (리다이렉트 X).

### API 매핑 (Phase I 완성분)

| 동작 | HTTP | 엔드포인트 | body |
|------|------|-----------|------|
| 목록 | GET | `/api/admin/mentor?status={PENDING\|APPROVED\|REJECTED}` | — |
| 상세 | GET | `/api/admin/mentor?status=...` 결과에서 id로 find (현재 상세 엔드포인트 없음) | — |
| 승인 | POST | `/api/admin/mentor/{id}/approve` | — |
| 반려 | POST | `/api/admin/mentor/{id}/reject` | `{reason: string}` (10~500자) |

> ⚠ **갭 1**: 현재 백엔드에 **단건 조회 엔드포인트가 없음** ([AdminMentorController.java](../../backend/src/main/java/com/devmatch/controller/AdminMentorController.java) 는 `GET /` 만 존재). 상세 페이지는 목록 응답을 클라이언트에서 캐시해 찾는 방식으로 일단 구현하고, 다음 백엔드 이터레이션에서 `GET /api/admin/mentor/{id}` 추가 예정. 사용자 확인 필요.

### 상태 필터 규칙

- **기본 진입** → `PENDING` 탭 선택 + 배지에 숫자 (조회 후 계산)
- 탭 전환 시 URL 쿼리 업데이트 (`?status=APPROVED`) → 새로고침 대응.
- 검색은 클라이언트 사이드 필터링 (이름 + 이메일 부분 일치). 백엔드 검색 지원은 다음 이터레이션.

## 3. 반려 모달 (Reject Dialog)

> [반려] 버튼 클릭 시 중첩 Dialog 오픈.

```
┌────────────────────────────────────────────────────────────┐
│  멘토 신청 반려                                        [×] │  DialogHeader + DialogTitle
│  김멘토 님의 신청을 반려합니다.                            │  DialogDescription
├────────────────────────────────────────────────────────────┤
│                                                            │
│  반려 사유 *                                               │  Label
│  ┌────────────────────────────────────────────────────┐    │  Textarea (rows=5)
│  │ 신청자에게 안내할 사유를 10~500자로 입력하세요.    │    │  required, min=10, max=500
│  │                                                    │    │
│  │                                                    │    │
│  └────────────────────────────────────────────────────┘    │
│                                         12 / 500 (최소 10) │  카운터 (muted)
│                                                            │
│  ⓘ 입력한 사유는 신청자에게 노출됩니다.                   │  Alert info
│                                                            │
├────────────────────────────────────────────────────────────┤
│                         [ 취소 ]       [ 반려 확정 ]       │  DialogFooter
│                      variant="ghost"   variant="destructive"│
└────────────────────────────────────────────────────────────┘
```

### 유효성 & 동작

- 10자 미만이거나 500자 초과면 `[반려 확정]` 비활성화.
- 확정 클릭 → `POST /api/admin/mentor/{id}/reject` → 성공 시 모달 닫고 목록으로 리다이렉트 + `toast.success`.
- 서버 400 응답 시 모달 안에 에러 표시 (모달 유지).

### shadcn 컴포넌트 매핑

| UI 요소 | shadcn 컴포넌트 | 비고 |

| 모달 | `Dialog` + `DialogContent/Header/Title/Description/Footer` | **`npx shadcn@latest add dialog` 필요** |

## 상태 관리 / 데이터 흐름

- **SWR 또는 TanStack Query** 중 기존 코드와 맞추기 → 현 프로젝트는 `fetch` + `useEffect` 기반(확인 후 결정). 일단 `fetch` + 로컬 state 유지.
- 목록 캐시: `/admin/mentor?status=X` 키로 status별로 분리 저장, 상세는 해당 캐시에서 `id === params.id` 로 검색.
- 승인/반려 성공 → 해당 status 캐시 무효화 후 목록으로 리다이렉트.
- 토스트는 `sonner` 선호 (shadcn 권장).

## 결정 사항 (2026-04-21 확정)

1. **상세 엔드포인트** — **목록 캐시 재활용 방식으로 진행**. 상세 페이지 진입 시 해당 status 의 목록을 먼저 호출(없으면) 후 `id` 매칭. `GET /api/admin/mentor/{id}` 단건 엔드포인트는 **다음 백엔드 이터레이션에서 보강 예정** (후속 이슈로 관리).
2. **페이지네이션** — **이번 범위에서 제외**. 백엔드 페이징 미지원이라 클라이언트 사이드 페이징도 생략. 데이터가 20건을 넘기 시작하면 서버 페이징 추가와 함께 도입. 목록 테이블 아래에 `TODO` 주석 남김.
3. **토스트** — **sonner** 사용 (shadcn 공식 권장).
4. **신규 shadcn 컴포넌트** — 설치 진행. `tabs`, `table`, `dialog`, `sonner`. (pagination 은 2번 결정으로 제외.)
5. **관리자 콘솔 시각 차별화** — **적용**. Header 배경을 neutral 톤(zinc-100 또는 slate-100)으로 두고 좌측에 "관리자 콘솔" 라벨. 좌측 Sidebar 는 이번 범위에서는 간단한 세로 네비(멘토 심사 한 개)로 렌더.
