# Phase II UI — 관리자 게시물 관리 목업 (Markdown 버전)

> `admin-mentor.md` 와 동일한 포맷. 커뮤니티 게시물 조회 + 강제 삭제.

---

## 공통

### 접근 제어

- `/admin/posts/**` 경로는 **ADMIN 역할**만. `SecurityConfig.java:49` 공통 가드.

### 라우팅

| 경로 | 화면 |
|------|------|
| `/admin/posts` | 게시물 목록 (카테고리·기간 필터) |
| `/admin/posts/[id]` | 게시물 상세 + 강제 삭제 / 댓글 관리 |

### API 매핑 (전부 신규 백엔드 작업 필요)

| 동작 | HTTP | 엔드포인트 | body | 현황 |
|------|------|-----------|------|------|
| 목록 | GET | `/api/admin/posts?category=&q=&from=&to=&page=&size=` | — | ❌ 없음, 신규 |
| 상세 | GET | `/api/admin/posts/{id}` | — | ❌ 없음, 신규 (사용자용 `GET /api/posts/{id}` 재사용도 가능) |
| 강제 삭제 | DELETE | `/api/admin/posts/{id}` | `{reason: string}` | ❌ 없음, 신규 |
| 강제 댓글 삭제 | DELETE | `/api/admin/posts/{postId}/comments/{commentId}` | `{reason: string}` | ❌ 없음, 신규 |

> ⚠ **갭 1**: 기존 `PostController.java:78` 의 `DELETE /api/posts/{postId}` 는 **작성자 본인만** 삭제 가능. 관리자 강제 삭제는 별도 엔드포인트 필요.

> ⚠ **갭 2**: `Post` 엔티티에 `deleted`, `hiddenByAdmin`, `deletionReason` 필드 **없음**. 하드 삭제로 갈지 소프트 삭제를 추가할지 결정 필요 (§ 결정 필요 사항).

> ⚠ **갭 3**: 신고(report) 기능이 현재 **없음**. Phase II 목업은 "관리자가 직접 발견/확인하고 삭제" 시나리오로 작성. 신고 기능은 후속 이터레이션.

### 카테고리

- `Post.category` 는 free-form String 이지만 시드/운영에서 사용되는 값: `질문`, `후기`, `정보`, `잡담` 등.
- 목록 필터는 `Select` 로 제공, 기본 "전체".

- 성공 시 목록으로 리다이렉트 + `toast.success("게시물이 삭제되었습니다")`.

## 결정 필요 사항

1. **하드 삭제 vs 소프트 삭제**
   - (a) **하드 삭제**: 물리 삭제. 단순하지만 감사 로그와 병행 필요.
   - (b) **소프트 삭제**: `Post.deleted: boolean`, `deletionReason`, `deletedBy`, `deletedAt` 컬럼 추가. 사용자 조회 시 필터링 로직 모든 곳에 추가 필요.
   - → **권장: 소프트 삭제 + 사용자 조회 쿼리에 `deleted = false` 기본 필터**.
2. **신고 기능** — Phase II 포함 여부. 포함하면 `PostReport` 엔티티 신설 + 작성자에게 가림 처리 워크플로.
3. **작성자 알림** — 삭제 시 이메일/푸시 발송 여부.
4. **카테고리 표준화** — 현재 free-form String → Enum 화할지? (별도 이슈)
5. **댓글 소프트 삭제** — `Comment` 엔티티에도 동일 정책 적용 필요.
