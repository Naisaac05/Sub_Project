# AI Inference Orchestration 운영 리허설 / 릴리즈 게이트

> 작성일: 2026-05-27  
> 대상: DevMatch AI Review local LLM inference orchestration  
> 목적: P0~P3 운영 보강 작업을 pre-prod 릴리즈 전에 한 번에 검증하는 실행 게이트

---

## 1. 게이트 원칙

이 문서는 production 전환 선언을 위한 문서가 아니라, **릴리즈 후보가 운영 리허설을 통과했는지 판정하는 절차**다.

- 모든 smoke command는 같은 커밋/같은 설정 묶음에서 실행한다.
- 실패한 항목은 원인과 조치가 기록되기 전까지 통과 처리하지 않는다.
- degraded mode, Redis, gateway, semantic eval, full RAG eval은 각각 따로 켜고 끄며 확인한다.
- 운영 profile에서 timeout, token, bounded stream/input, JSONL 금지 조건을 반드시 확인한다.
- 리허설 결과는 날짜별 evidence로 남긴다.

## 3. Pre-Flight 환경 점검

### 3.1 Production Config Gate

운영 profile에서는 아래 조건이 모두 true여야 한다.

- Spring service token이 비어 있지 않다.
- Python service token이 비어 있지 않다.
- Spring stream timeout이 양수다.
- Python Ollama request timeout이 양수다.
- queue wait timeout이 양수다.
- stream/input upper bound가 존재한다.
- candidate sink가 JSONL source-of-truth로 동작하지 않는다.
- `OLLAMA_REQUEST_TIMEOUT_SECONDS=0` 또는 음수 설정이 없다.

판정:

```text
PASS: startup validation이 성공하고 위 조건이 모두 확인됨
FAIL: 하나라도 누락, 0/음수, blank, JSONL 운영 source-of-truth가 발견됨
```

### 3.2 Runtime Dependency Gate

확인 항목:

- Redis 연결 가능
- Ollama endpoint health 확인
- model pool endpoint 중 draining 대상 제외 확인
- vector index manifest 읽기 가능
- answer cache namespace가 active index와 일치

## 4. Smoke Command Set

아래 명령은 `C:\Users\User\Desktop\Sub_Project\ai` 기준으로 실행한다.

전체 릴리즈 게이트는 아래 명령으로 한 번에 실행할 수 있다.

```powershell
.\.venv\Scripts\python.exe scripts\run_release_gate.py --output release-gate-report.json
```

스크립트는 각 smoke command의 exit code, stdout/stderr, 실행 시간, RAG evaluator threshold 판정, 최종 `GO` / `NO-GO` / `CONDITIONAL-GO` 결정을 JSON으로 저장한다.

명령만 점검하려면 dry-run을 사용한다.

```powershell
.\.venv\Scripts\python.exe scripts\run_release_gate.py --dry-run --output release-gate-dry-run.json
```

## 5. Degraded Mode 리허설

각 kill switch는 단독으로 켜고, 리허설 후 반드시 원복한다.

### 5.1 streaming_off

기대 동작:

- Spring streaming path가 비활성화된다.
- sync/fallback path가 응답한다.
- `fallback_to_sync_count` 또는 대응 metric이 증가한다.

### 5.2 cache_only

기대 동작:

- cache hit은 정상 응답한다.
- cache miss는 Ollama 호출 없이 cache-only miss 응답을 반환한다.
- model generation count가 증가하지 않는다.

### 5.3 template_fallback_only

기대 동작:

- Ollama 호출 없이 template answer가 반환된다.
- `fallback_used=true`
- route가 template fallback 계열로 기록된다.

### 5.4 lightweight_only

기대 동작:

- static/generated-card fast path는 정상 응답한다.
- miss는 `route=lightweight_only_miss`
- `llm_call_avoided` 품질 플래그가 기록된다.

### 5.5 no_candidate_capture

기대 동작:

- answer path는 정상 응답한다.
- candidate 저장이 발생하지 않는다.
- `candidate_capture_disabled` 품질 플래그가 기록된다.

## 6. Load / Failure 리허설

아래 runner는 local dev-laptop 기준의 반복 가능한 부하 시나리오다.

```powershell
.\.venv\Scripts\python.exe scripts\run_stream_load_profile.py --scenario cache-hit
.\.venv\Scripts\python.exe scripts\run_stream_load_profile.py --scenario cache-miss
.\.venv\Scripts\python.exe scripts\run_stream_load_profile.py --scenario free-form-storm
.\.venv\Scripts\python.exe scripts\run_stream_load_profile.py --scenario stream-disconnect
.\.venv\Scripts\python.exe scripts\run_stream_load_profile.py --scenario ollama-timeout
```

통과 기준:

- timeout/hang이 bounded time 안에서 종료된다.
- disconnect 시 terminal state는 `DISCONNECTED` 하나만 남는다.
- completed/error/disconnected 중 하나만 terminal state로 저장된다.
- semaphore leak이 없다.
- degraded mode 전환 후 answer path가 막히지 않는다.

## 7. Metrics / Dashboard Gate

리허설 동안 최소 아래 metric이 확인되어야 한다.

Stream lifecycle:

- `first_token_latency_ms`
- `stream_duration_ms`
- `stream_completed`
- `stream_disconnected`
- `stream_partial_failed`
- `fallback_to_sync_count`

Ollama / gateway:

- model
- endpoint
- capacity
- in_flight
- queue wait
- timeout count
- semaphore_released

Workflow / quality:

- route
- fallback_used
- quality_flags
- candidate_capture_disabled
- candidate_capture_failed
- semantic grounding metrics
- hallucination cache-ban rate

판정:

```text
PASS: metric sink 또는 dashboard에서 위 항목을 조회할 수 있음
FAIL: 리허설 결과를 설명할 metric이 누락됨
```

## 9. Rollback / Kill Switch 우선순위

장애 시 우선순위:

1. `template_fallback_only`
2. `cache_only`
3. `streaming_off`
4. `lightweight_only`
5. `no_candidate_capture`
6. 이전 vector index manifest rollback
7. 이전 cache namespace rollback
8. gateway draining endpoint 조정

원칙:

- 사용자 응답 path 보호가 1순위다.
- candidate capture와 promotion은 answer path보다 낮은 우선순위다.
- semantic/eval 실패는 cache write와 promotion을 막는 방향으로 처리한다.
